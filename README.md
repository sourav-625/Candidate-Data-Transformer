# Candidate Data Transformer Pipeline

An end-to-end data ingestion and processing pipeline designed to parse, normalize, and merge candidate profile data from both structured and unstructured sources into a unified, canonical format.

The pipeline is built with a strict engineering principle in mind:

> **"A wrong-but-confident value is worse than a missing value."**

It utilizes deterministic rule-based matching alongside Natural Language Processing (NLP) to maximize extraction accuracy and resolve data conflicts safely.

## 🛠️ System Architecture & Data Flow

The pipeline processes data through three distinct layers: **Ingestion & Parsing**, **Normalization**, and **Merging & Resolution**.

```

                       [ CSV Data ]          [ Recruiter Notes ]
                            │                        │
                            ▼                        ▼
                     ┌──────────────┐        ┌──────────────┐
                     │ csv_parser.py│        │notes_parser.py│
                     └──────┬───────┘        └───────┬──────┘
                            │                        │
                            └───────────┬────────────┘
                                        ▼
                               [ Candidate Objects ]
                                        │
                                        ▼

                                        ▼

                           └────────────┬─────────────┘
                                        ▼
                               [ Canonical Output ]

                               
```

## 1. Ingestion & Parsing Layer

### `csv_parser.py`
Handles structured candidate data by deterministically mapping incoming fields to the internal schema using a configurable `field_mapping.json` resource file.

---

### `notes_parser.py`
A modularized parser that breaks down unstructured recruiter text using raw regular expressions and an NLP pipeline (`en_core_web_sm`). Rather than relying on a monolithic extraction block, it isolates responsibilities into independently testable functions:

#### `extract_name()`
Extracts names using Named Entity Recognition (NER) with a reliable regex fallback sequence.

#### `extract_location()`
Classifies geographical entities into city, state, and country keys via lookup heuristics instead of dumping all location data into a single field.

#### `extract_companies()`
Filters out false positives (e.g., when NLP models misclassify technologies like SQL or TensorFlow as organizations) by validating that the organization entity appears within an explicit employment context sentence.

#### `extract_skills()`
Implements context-aware extraction by segmenting text clauses and checking for negative qualifiers (e.g., *"weak in"*, *"lacking"*, *"no experience"*), actively rejecting skills the candidate does not possess.

---

## 2. Normalization Layer

### `normalization/`
Contains shared logic used by both parsers to ensure consistency across extracted fields before merging.

Examples:
- Phone number formatting into a standard schema
- Skill alias normalization (e.g., `"Python3"` → `"Python"`, `"ML"` → `"Machine Learning"`)

This ensures downstream merging operates on clean, canonical data.

---

## 3. Merging & Resolution Layer

### `merger.py`
Consolidates duplicate or partial candidate records generated by separate ingestion paths into unified `Candidate` objects.

---

### `conflict_resolver.py`
Evaluates overlapping fields and safely determines field-level precedence.

Key design characteristics:
- Strict data-safety rules
- Avoids brittle operator constraints like `!=`
- Explicit, deterministic resolution logic for each field type
- Ensures no unsafe overwriting of high-confidence data

## 🚀 Getting Started

### Prerequisites
- Python 3.8+
- spaCy English small model (`en_core_web_sm`)

---

### Installation & Setup

Clone the repository and navigate to the project root:

```bash
git clone <repository-url>
cd candidate-data-transformer
```

### Install dependencies:

```
pip install -r requirements.txt
```

### Download the required spaCy model:

```
python -m spacy download en_core_web_sm
```

### Running the Pipeline

Execute the primary driver script:

```
python main.py --csv data/input/candidates.csv --notes data/input/recruiter_notes.txt --config configs/default.json --output data/output/candidate.json
```

# 🎯 Key Design Features

## Context-Aware Skill Filtering

Prevents false data ingestion. For example:

"Strong Python developer but weak in Java"
Correctly extracts Python while discarding Java.

## Targeted Entity Disambiguation

By establishing strict WORK_KEYWORDS boundaries, the pipeline avoids misclassifying technical terms as company entities.

## High Scannability & Maintainability

The modular architecture allows developers to update individual extractors (e.g., extract_location) without introducing regression risk across the pipeline.
